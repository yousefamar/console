## Architecture
- **PWA** — installable standalone, works in any browser including mobile
- **Android APK** (`android/`) — thin native-Kotlin WebView wrapper for the phone (PWA install was flaky). Loads `https://con.amar.io/`. Routes OAuth into Chrome Custom Tabs, handles `console://` deep links (incl. `?ott=…` cookie-jar handoff for the Custom-Tab → WebView OAuth path), in-app updater polls `/public/apk/latest.json`.
- **Public SPA + public hub via Caddy.** `con.amar.io` resolves to the home WAN IP (Namecheap DDNS — see Al's `ddns-update.md` workflow), Caddy on :443 with a Let's Encrypt cert serves: `/hub/*` → `https://127.0.0.1:9877` (Caddy strips `/hub`, sets `X-Forwarded-Host/Proto/-Prefix`); `/public/*` → `https://127.0.0.1:9877` (prefix preserved, never auth-gated); legacy `/cron.ics` rewrites to `/public/cron.ics`; legacy `/apk/*` rewrites to `/public/apk/*` (for stuck pre-v21 installs); catch-all → `http://127.0.0.1:5173` (Vite). Hub binds `127.0.0.1:9877` — Caddy is the only external ingress; emergency override is `--host 0.0.0.0`. The global Caddy block forces `protocols h1` on `:443` because HTTP/2 → HTTP/1.1 WebSocket bridging was unreliable for fresh upgrades (chrome timed out on new wss opens). SPA's `getHubUrl()` returns `${window.location.origin}/hub`. **Different origin = different IndexedDB** — one-time cross-origin migration UI in `AccountModal` ("Import from another origin"), powered by `src/migration.ts` (hidden iframe + postMessage pump). See `memory/project_public_funnel_arch.md` and `memory/reference_caddy_funnel_setup.md`.
- **Hub auth — cookie + bearer.** Single allow-list (`hubAllowedEmails` in `~/.config/console/auth.json`, default `['yousefamar@gmail.com']`). Browser path: SPA hits `${origin}/hub/auth/google/start?return=…`, hub does Google OAuth, callback sets `console_session=<id>; HttpOnly; Secure; SameSite=Lax; Path=/hub`. Sessions stored in `auth.json` `hubSessions[]`, 30-day TTL. **APK Custom Tab → WebView cookie handoff** via one-time token: hub callback redirects to `console://auth/done?ott=<ott>`; the APK navigates the WebView to `/hub/auth/claim?ott=…` which validates the OTT (single-use, 2-min TTL) and issues a fresh `Set-Cookie` in the WebView's jar. Non-browser path: named bearer tokens (`hubTokens[]`, hashed sha256, individually revocable) sent as `Authorization: Bearer <…>`. Scopes: `cli`, `al`, `apk`, `other`. CLI + Al read plaintext from `~/.config/console/local-tokens.json` (mode 0600) — auto-minted on hub first boot, separate per scope. APK gets paired via SPA "Pair this APK" → `POST /hub/auth/tokens` → `window.ConsoleNative.setHubToken(…)` → `HubTokenStore.kt` (EncryptedSharedPreferences). PushService attaches the bearer via OkHttp `Request.Builder().header("Authorization", …)` and posts a "needs re-pair" notification on close-4401/4403 or HTTP 401/403 during handshake. **Enforcement** gated by `CONSOLE_AUTH_ENABLED=1`; **break-glass** is `CONSOLE_AUTH_DISABLED=1` (forces allow even if enabled). Both env vars set in pm2's environment. The loopback bypass requires NO `X-Forwarded-*` headers (so Caddy-proxied requests aren't silently treated as same-host). Mutating session-authed requests also pass a `SameSite=Lax`-belt-and-braces `Origin`/`Referer` check (`isCsrfSafe`). Confused-deputy guard: requests carrying BOTH cookie and bearer are rejected. **Pre-auth render gate**: `src/GatedBoot.tsx` is the SPA's only entry point — it probes `/hub/auth/session` first and renders LITERALLY nothing until authenticated; the rest of the app (`App`, sync-bus, stores, panes) is lazy-imported behind the gate so an unauthenticated visitor can't even see pane scaffolding via DOM inspection.
- **Public canvas surface.** Each tab/island can be flipped public via `con dashboard canvas {tab,} publish <slug>` (or the Share button on the Home canvas card → `src/components/home/CanvasShareMenu.tsx`) → mints a 256-bit base64url token, persisted at `~/.config/console/canvas-public-tokens.json` (sibling of `canvas/` so `con dashboard canvas reset` does NOT silently revoke shares — only explicit `unpublish` does). Public URL: `https://con.amar.io/public/canvas/<token>/`. Routes in `server/src/routes/public.ts`: trailing-slash 301, asset path-traversal guard (`path.resolve` ⊂ `tabsDir/<slug>/`), `no-store`/`nosniff`/`no-referrer` HTML headers. Islands are inline-HTML only (no asset paths); tabs serve files from their dir.
- **Offline-first** — all mutations (email, chat, calendar) write to IndexedDB first, sync via queue
- **Hub server** — Node.js backend (`server/`) proxies all API calls (Gmail, Calendar, Matrix, Monzo), manages OAuth tokens, hosts agent sessions. Internal HTTPS via any `*.crt`/`*.key` pair in `~/.config/console/` (Caddy terminates the public TLS in front). Binds `127.0.0.1:9877` — all external traffic goes through Caddy.
- **Sub-app isolation** — Layout subscribes only to `activePane`. Each tab component is `React.memo`'d. A chat state change never re-renders the email pane.
- **Pre-rendered panes** — all panes mounted with `display:none`, toggled on selection for instant switching. `EmailFrame` renders a height-preserving placeholder when `!visible` (fixed 2026-07-02): dozens of live hidden email iframe documents used to fire recurring ~500ms browser-internal long tasks that lagged typing in OTHER panes (root cause of the Agents-tab input-lag recurrence). See `memory/project_emailframe_iframe_perf.md`.
- **Hub WS heartbeat** — `src/sync-bus.ts` runs a 15 s ping (server already responds at `server/src/sync-bus.ts:95`). 30 s without inbound traffic → `forceReconnect()`. Catches the APK / mobile-Safari "WS half-dead while backgrounded" bug at the source — no visibility-event dependence. See `memory/project_apk_ws_heartbeat.md`.

## Debugging (PRIMARY METHOD)

**Use the embedded debug agent, NOT browser DevTools or Chrome MCP tools.**

The app has a built-in debug agent (`src/debug-agent.ts`) that streams all console output, network requests, and errors to the hub in real-time. Claude Code observes via HTTP endpoints:

```bash
# See recent events (console logs, network requests, errors)
curl -sk https://localhost:9877/debug/log?n=30

# Filter for errors
curl -sk https://localhost:9877/debug/log?n=100 | jq '.[] | select(.cat=="error")'

# Filter for failed network requests
curl -sk https://localhost:9877/debug/log?n=100 | jq '.[] | select(.cat=="net" and .status >= 400)'

# Execute JS in the browser (replaces Chrome MCP javascript_tool)
curl -sk -X POST https://localhost:9877/debug/eval -H 'Content-Type: application/json' -d '{"code":"document.title"}'

# Get Zustand store state
curl -sk https://localhost:9877/debug/state

# Check if debug agent is connected
curl -sk https://localhost:9877/debug/status
```

The debug agent:
- Captures from page load (imported first in `main.tsx`)
- Survives navigation (WebSocket auto-reconnects)
- Works across all tabs/devices (all stream to same hub)
- Logs to `~/.config/console/debug.log` (NDJSON, auto-rotated at 5K lines)
- `window.__console` (dev-only) exposes Zustand stores + Dexie db for browser console access

## Key Patterns

### Auth
- **Hub-managed OAuth** — all tokens stored server-side in `~/.config/console/auth.json`. Frontend bootstraps via `GET /auth/token`. Sign-in opens hub's `/auth/google/start` in a popup.
- **Multi-account** — hub stores multiple Google accounts. Calendar sidebar can add accounts via OAuth flow.
- **Token refresh** — hub proactively refreshes tokens 5min before expiry. Frontend re-fetches from hub on 401.

### Offline Sync Queue
- **Shared queue** — `src/db/sync-queue.ts` provides `enqueue()`, `markDone()`, `markFailed()` for all services
- **Action types** — email (`archive`, `send`, etc.), chat (`chatSend`, `chatMarkRead`), calendar (`calCreate`, `calUpdate`, `calDelete`, `calRsvp`, `calReminder`, `calLocation`)
- **Pattern** — optimistic IDB write → `set()` for instant UI → `enqueue()` → background processor flushes within 500ms
- **Retry** — 3 retries before marking `failed`. Queue survives page reload.
- **Undo** — delete actions show 5-second undo toast. Undo removes from queue + restores IDB.

### Email
- Optimistic archive/snooze with undo. Auto-archive on reply. Pre-rendered iframes. CID inline images.
- Snooze is local-only (Gmail has no snooze API). Conflict detection before sending queued replies.
- Send-as aliases fetched during sync. Calendar invites parsed from `text/calendar`.

### Chat (Matrix)
- **Hub-owned room snapshot.** The canonical view of every joined room (name, isUnread, unreadCount, avatar, memberCount, tags, snoozedUntil, lastMessage*) lives server-side in `server/src/matrix/chat-rooms-store.ts` (a `SnapshotStore` wrapper from `server/src/snapshot-store.ts`). Every `/sync` delta computes the row via `server/src/matrix/room-state.ts` (port of the SPA's old `processJoinedRoom` logic) and the store broadcasts on the `chat-rooms` service. Browser-side `src/matrix/chat-rooms-subscribe.ts` mirrors snapshot + deltas straight into `db.chatRooms`. Replaces per-device derivation — "mark read on PC → phone sees it" works without local divergence. RPCs registered under `chat-rooms`: `snapshot`, `markRead`, `markUnread`, `snooze`. **Manual unread is durable** (`RoomState.manualUnread`): a user "mark unread" has no homeserver concept, so `computeRoomState` previously cleared it on any sync reporting `notification_count=0` — and the big catch-up `resume`/sync after a hub restart touches enough rooms to wipe *all* manual markers at once. The sticky flag forces `isUnread` true across recomputes and is cleared only by an explicit `markRead` (`setRoomRead`); genuine notif-backed unread was never affected.
- Direct REST API (no matrix-js-sdk). E2EE via OlmMachine WASM. Long-polling sync.
- Local echo with `~timestamp.random` IDs. Bridge detection (WhatsApp, LinkedIn, Slack).
- Pre-rendered room views. Pinned favourites grid. Edit/delete diffs via `diffWords`.
- **Send failures surface in the UI**: Beeper bridges emit `com.beeper.message_send_status` events referencing the original encrypted event. Browser-side `src/matrix/sync.ts` flips `sendFailed` on the local echo and fires a notification. SUCCESS status clears any prior fail marker. Root cause of FAIL_RETRIABLE is almost always the hub device not being cross-signed — run `POST /matrix/keys/restore-cross-signing` (see Matrix recovery workflow below).
- **Force-rotate Megolm**: `matrix.rotateRoomKey({roomId})` RPC (→ `HubMatrixCrypto.invalidateRoomKey`) throws away the outbound session so the next send re-keys from scratch. Useful when a bridge gets into a wedged state. Note: `invalidateGroupSession` only touches the in-memory session, so we pre-load by calling `shareRoomKey(roomId, [], …)` first — otherwise after a hub restart the persisted stale session stays live.
- **Resume semantics (gap recovery)**: the hub runs one /sync loop and broadcasts `matrix.delta` over SyncBus — fire-and-forget, no per-client buffer. Reconciliation uses Matrix's **native** resumable-sync primitive: the client persists `next_batch` from every ingested delta (in `meta` table under `matrix:lastBatch`), and hands it back to the hub via `matrix.resume({since})` RPC on any wake signal. Hub calls `/sync?since=<token>&timeout=0`, decrypts, returns the delta directly (point-to-point, not broadcast). Trigger points all converge on a single debounced `reconcile()` in `src/hooks/useSync.ts`: (1) `hubBus.onConnect`, (2) `document.visibilitychange → visible` (critical for APK — the WebView pauses while backgrounded so WS stays "up" from JS's perspective and onConnect never fires), (3) `window.online`. `bulkPut` on event_id makes re-ingestion idempotent, so racing live-delta + resume is safe. If the homeserver rejects the `since` (expired), hub falls back to initial sync and flags `isInitial` on the response. `processJoinedRoom` gates `lastMessageBody`/`lastMessageTime` behind `advancesPreview` (newest-by-timestamp > existing) so out-of-order resume ingestion can't roll a room preview back to an older message.
- **Limited-timeline backfill**: Matrix `/sync` caps each room's timeline at ~10 events and sets `limited: true` + `prev_batch` when more happened during the gap. The APK WebView gets paused while backgrounded — on resume, its `since` token is far behind and sync returns many truncated rooms, so without backfill the local cursor jumps past real messages. Hub-side `backfillLimitedTimelines()` in `server/src/matrix/sync.ts` walks `/rooms/{id}/messages?from=prev_batch&dir=b` (limit 100) for each limited room, reverses to forward order, decrypts, and **prepends** the gap events before broadcasting the delta. Runs on both `resume()` (skipped on `isInitial`, since the full initial sync has no gaps) and `tick()` (live loop). Idempotent via `bulkPut(event_id)` client-side.
- **Redaction tombstones (bridges)**: WhatsApp / Signal delete events come through as `m.room.encrypted` with `content: {}` (the original ciphertext stripped) plus `unsigned.redacted_because.{sender,type:"m.room.redaction"}`. Without special handling they render as "🔒 Encrypted message" placeholders. Fix lives in `src/matrix/sync.ts`: `eventToMessage` returns `null` for tombstones, and `processJoinedRoom` has a pre-loop guard that flips any existing row's `isDeleted=true` + `deletedBy=<redactor>` when a tombstone for it arrives. Detect via `event.type === 'm.room.encrypted' && typeof event.content?.ciphertext !== 'string'`.
- **Soft-delete-only guarantee (message archive)**: Console NEVER hard-deletes chat content — deletes by any party only ever mark rows `isDeleted`; original text + media stay recoverable. `server/src/matrix/message-archive.ts` (`MessageArchive`, `~/.config/console/chat-archive/`): append-only NDJSON per room (`events/<sha1(roomId)>.ndjson`, dedup-by-first-record so the pre-redaction copy wins) + `media/<sha1(eventId)>.bin` decrypted-attachment rescue. Wired in `MatrixSync.decryptJoinedRoomsWithCounts` (live sync) and `paginate` (history): every decrypted message-bearing event is archived BEFORE downstream processing; observing a redaction (explicit `m.room.redaction` or encrypted tombstone) triggers `recordRedaction` → downloads + AES-CTR-decrypts the attachment to disk (blobs outlive event redaction on Beeper). The archive class has deliberately NO delete/prune API. Surface: `chat-rooms.archivedEvent {roomId,eventId}` RPC (pre-redaction copy) + `GET /matrix/archive/media/<sha1>.bin`. SPA `DeletedMessageBody` in `ChatMessageBubble.tsx` renders deleted messages by falling back to the archive (struck-through original text, recovered image inline). Limit: content the hub never saw (deleted before Console ever synced the room) is unrecoverable — the homeserver strips it.
- **@-mention autocomplete** (`src/components/ChatComposeInput.tsx`). `@<query>` only opens the picker when the `@` is at start-of-input or after whitespace (so emails like `alice@example.com` don't trigger). Surviving `@DisplayName` tokens at send time become matrix.to anchor tags in `formatted_body` and userIds in an `m.mentions` block per MSC3952 — Element / Cinny / bridges treat the message as an intentional ping. Deleting the `@token` before send removes the userId from the m.mentions block. Room-members cache lives in `src/matrix/room-members.ts` (module-level, primed via `/matrix/rooms/:id/info`).
- **Hub Megolm backup upload (auto)**: on every boot, `server/src/index.ts` calls `hubMatrixCrypto.activateBackupUpload(homeserver, token)` which GETs `/room_keys/version`, extracts the public key, and calls `enableBackupV1(pub, ver)` — encryption-only, no recovery key needed. From that point on every new Megolm session the hub receives via to-device is auto-uploaded through `processOutgoingRequests` → `/room_keys/keys`, so a later hub re-init can restore everything. A one-shot `backupPendingRoomKeys()` sweep runs after activation to upload any sessions imported from the local M0 backup or received while backup was inactive. Boot log line: `[hub-crypto] backup upload activated (version N)` followed by `[hub-crypto] backed up N pending room keys` if any were outstanding.

### Calendar
- **Hub-proxied** — all API calls go through hub (`/cal/*`), which handles tokens server-side
- **Offline-first** — all mutations (create, update, delete, RSVP, reminder, location) are optimistic with sync queue
- **Temp IDs** — created events use `~timestamp.random` IDs, replaced with real Google IDs after queue processes
- Custom week/day grid. Drag-to-create/move/resize. Recurring event dialog. Event merging across calendars.
- **Read-only overlay sources** (`useCalendarStore.registerOverlaySource(id, info, events)` / `unregisterOverlaySource(id)`): a generic seam for showing non-Google, non-persisted events in the grid. The `CalendarInfo` must carry `synthetic: true` (skipped by `fetchEvents` so there's no bogus Google fetch for a fake account) + `accessRole: 'reader'` (gates the grid drag/resize + popover edit/delete — all keyed on accessRole, never on a specific id). Events are merged in `loadEventsFromDb` (same range+visibility filter as Dexie events), live only in memory (`overlaySources`), and the source's `id` must be unique per event (no Dexie compoundKey). First registration defaults the source visible (`OVERLAY_SEEN_PREF`). **GOTCHA: `OVERLAY_SEEN_PREF` and `VISIBLE_CAL_IDS_PREF` must be written ATOMICALLY** — both gated on `isPrefsLoaded()`. Marking seen without persisting visibility (the pre-2026-07-09 behaviour, hit when a register raced `initPrefs()` at boot) left the overlay seen-but-absent-from-visibleIds → treated as user-hidden → its sidebar toggle silently vanished on every subsequent boot. `fetchCalendars` also re-asserts overlay visibility after reloading the saved pref. All synthetic infos group under ONE "Overlays" bucket in `CalendarSidebar` (keyed `' overlays'`, not per fake accountEmail). Consumers: **Meetup** (`src/meetup/calendar-overlay.ts` — `meetupEventToCalendarEvent` + `wireMeetupCalendarOverlay()` subscribing to `useMapStore.events`) and **OutdoorLads** (below).
- **OutdoorLads events (calendar-only overlay)**: hub fetches the public national RSS feed `https://www.outdoorlads.com/events-rss.xml` (`server/src/outdoorlads.ts` `OutdoorLadsStore`, 1h TTL cache, never background-polled; routes `/outdoorlads/{events,status}` in `server/src/routes/outdoorlads.ts`). The feed repurposes `<pubDate>` as the event START time and embeds Event Type / Region / Description as HTML in `<description>` — parsed with field regexes. **No geo coords** → no Map pin, calendar overlay only. Client `src/outdoorlads/calendar-overlay.ts` (wired in `GatedBoot`): fetches, filters to camping (`INCLUDE_TYPES = ['camp']`, hardcoded per Yousef), adapts to a synthetic "OutdoorLads" calendar (orange `#f5821f`), refreshes on boot (with retry backoff — a single failed fetch at boot must NOT give up; `onConnect` won't re-fire if the WS was already up) + hub-reconnect + 6h interval. Feed gives no end time → events render as 2h blocks.
- **Meetup client mirror is snapshot-authoritative** (`src/meetup/subscribe.ts` `fetchSnapshot`): rows absent from the hub snapshot are DELETED from Dexie + the map store (preserving loaded `detail` on survivors) — merge-only mirroring kept hub-pruned/cleaned events alive client-side forever (the "calendar full of Amsterdam events" bug; the hub-side store is append-only with expiry-only pruning, so a location cleanup = edit `meetup-events.json` + hub restart, and this authoritative snapshot propagates it).

### Bookmarks
- 977+ `.md` files in Obsidian vault. Hub parses + caches. Browse mode + Tinder-style triage mode.

### Notes
- CodeMirror 6 with vim mode. Obsidian-style live markdown preview. FSA or hub adapter.
- Quick Switcher (fzf fuzzy + MiniSearch full-text). Multi-file tabs. No auto-save.
- **View mode** (`useNotesStore.viewMode`, persisted via hub pref `notesViewMode`): `tree` (default `NotesFileBrowser` sidebar) | `circles` (`src/components/notes/CirclesView.tsx`). Toggle via the `Circle` icon in the tree toolbar / `FolderTree` icon in the circles toolbar.
- **Circles view** — canvas-based circle-pack visualization à la Machete/Dendron. `d3-hierarchy` packs `useNotesStore.files` weighted by `VaultFile.size` (added to the type for this; both adapters fill it via `fs.stat()` / `File.size`). Render: one `<canvas>`, rAF + dirty flag, all imperative. d3-zoom owns pan/wheel/pinch; its `filter` callback hit-tests at pointerdown and rejects gestures over files so they never race with drag-to-rename. Long-press 400ms (mobile + desktop) lifts a file for drag-to-move; quick tap fires the native `click` to open. Folders use Machete-style level-of-detail: nodes render in reverse depth order so each parent's opaque cover hides its children until the cover fades. The fade threshold (`coverFadeThreshold(W, H) = min(W,H) * 0.4`) scales with viewport — clicking a folder always zooms past it via `fitTransform(node, W, H, padding=1.05)`. `isAncestorChainOpen` gates both label rendering and hit-testing so labels can't leak through opaque covers (screen-space pass otherwise bypasses the painter's algorithm). Pure helpers live in `src/components/notes/circles-view-helpers.ts` with unit tests in `src/__tests__/circles-view-helpers.test.ts`.
- **Active-line highlight suppression during selection**: the default `highlightActiveLine` paints a line background that sits above CM6's selection layer, washing the selection out. Replaced with a custom `ViewPlugin` in `NotesEditorCore.tsx` that only adds the `.cm-activeLine` decoration when `state.selection.main.empty`, so any non-empty selection (visual mode, mouse drag) shows cleanly.
- **Tab nav** — `Ctrl+H` (prev) / `Ctrl+L` (next), wired as a capture-phase keydown listener on `window` in `useKeybindings.ts`. Capture phase is critical: it runs before CodeMirror's keymap (which would otherwise consume Ctrl+H as cursor-left in vim normal / `deleteCharBackward` in insert) AND lets `preventDefault()` stop Brave/Chromium's `Ctrl+H = history` accelerator (it's "unreserved" per Chromium's accelerator table — `Ctrl+T/N/W` are the reserved ones that can't be intercepted). Works in installed PWA + APK.
- **Blog writing mode** (third `viewMode: 'blog'`): `src/components/notes/BlogView.tsx` sidebar (New post, drafts age-coloured, projects with expandable devlogs, recent published with ↗ permalinks). When the active file is under `scratch/blog-drafts/` or `log/` ("writing files"): `WriteMetaBar` (title/tags/project structured editing — round-trips through the CM6 buffer via `src/utils/frontmatter.ts`, a client port of the server's parse/stamp, NEVER a server-side patch since the vault may be FSA-local) + `WriteActionBar` (Photo/Camera with canvas downscale to 2000px JPEG → `pasteImage`, Mic via `src/hooks/useDictation.ts`, Format via `POST /blog/format`, Publish/View-live morphing). Permalink formula: `log/<name>.md` → `https://yousefamar.com/memo/log/<name>/` (`permalinkForLogPath`).
- **Dictation hook** (`src/hooks/useDictation.ts`): extracted from AgentPromptInput — browser SpeechRecognition with fallback to hub `/stt` WS; APK goes straight to hub (WebView SR hangs silently). Commits text via `onText` per chunk; dedups VAD finals against streamed deltas. AgentPromptInput consumes it.
- **`POST /blog/format`**: dictation formatter — runs `claude -p` one-shot via the CLI (no ANTHROPIC_API_KEY needed, rides the subscription, model = CLI default). Strict prompt: punctuation/paragraphs/artefacts only, wording untouched. `GET /blog/posts?limit=N` lists recent published posts (title/date/project/tags).
- **Mobile vim**: off by default (`console:notes:mobileVim` localStorage); auto-enables on hardware-keyboard heuristic (real Escape / Ctrl-chord keydown — soft keyboards emit keyCode 229/'Unidentified'); manual toggle = tappable "vim" chip in the editor status bar.

### Money (Monzo + financial planner)
- Hub proxies Monzo API. Single-use refresh tokens. Server-side transaction cache (`monzo-transactions.json`, never deleted — Monzo's 90-day API wall is one-time pain at first sync).
- SCA required after OAuth. Webhook for real-time transaction notifications.
- **Sub-tabs in `MoneyTab.tsx`** (sub-tab state in `src/store/finance.ts`, persisted to localStorage `console:money:subtab`): `Cashflow | Net worth | Budgets | Scenarios | Categories | Transactions`. Default landing = Cashflow (runway-first); Transactions is the original 3-pane forensic view.
- **Financial-planning subsystem (`/finance/*`)** — independent from `/money/*`. All data persisted as JSON files under `~/.config/console/`:
  - `finance-categories.json` — user-defined (income/expense/transfer), seeded with 16 sensible defaults; replaces Monzo's 10 generic ones.
  - `finance-rules.json` — auto-categorisation: priority-ordered, conditions are AND-ed (`merchantContains`, `descriptionContains`, `counterpartyContains`, `amountSign`, `monzoCategoryEquals`). First match wins. Default rules map Monzo categories onto user categories.
  - `finance-tx-overrides.json` — per-transaction override (set category, mark ignore, mark transfer). Beats rules.
  - `finance-accounts.json` — Monzo (auto-balance) + manual accounts. Manual accounts have a balance ledger (`{date, balancePence, note}[]`) — interpolated for "balance on date X". Liquidity tag (liquid|investment|illiquid) gates inclusion in runway. `isExternal: true` = held-by-someone-else (e.g. money in someone else's ISA — counts toward net worth, not directly drawable).
  - `finance-streams.json` — recurring income/expense (`monthly|yearly|weekly`, optional `dayOfMonth`/`monthOfYear`, `growthPctYoy` for compounding annual growth, `startDate`/`endDate`).
  - `finance-budgets.json` — per-category monthly target.
  - `finance-scenarios.json` — what-if editor: baseline + ordered list of `Delta`s (`addStream | modifyStream | terminateStream | oneOff | categoryAdjust | investmentGrowth`).
  - `finance-settings.json` — emergency fund (`{mode:'fixed'|'months', valuePence|months}`), `projectionHorizonMonths`, `investmentGrowthPct`.
- **Pure projection engine** (`server/src/finance/projection.ts`):
  - `effectiveCategory(tx, rules, overrides)` — single source of truth for "what category is this transaction".
  - `aggregateMonthlySpend()` → per-month `byCategory` (positive = outflow, negative = inflow). Skips ignored + transfers.
  - `trailingCategoryAverage(monthly, windowMonths=3)` — variable-spend forecast input.
  - `streamAmountForMonth(stream, 'YYYY-MM')` — handles cadence + compounding growth.
  - `project({startMonth, horizonMonths, openingLiquid, openingInvestment, streams, variableForecast, categories, emergencyFund, investmentGrowthPct, scenario})` → `MonthlyPoint[]`. Variable spend skips categories already covered by an active fixed (non-variable) stream — prevents double-counting rent.
  - `summariseRunway()` → `{monthsToFloor, floorDate, monthsToZero, ...}` — walks the trajectory; first month liquid drops below the emergency floor wins.
  - `detectRecurring(txns)` — clusters by (label, ±50p amount band), requires ≥3 occurrences across ≥3 months. Surfaced in CashflowView as "Detected recurring".
  - `findTransferCandidates(txns, overrides)` — opposite-sign equal-amount within 3 days; UI for confirmation pending.
  - `manualBalanceLookup(account)` — latest entry on or before date; before-first-entry returns earliest known.
- **Routes** (`server/src/routes/finance.ts`):
  - CRUD: `/finance/{categories,rules,accounts,streams,budgets,scenarios,overrides,settings}` (POST = create, PATCH `/:id` = update, DELETE `/:id`). Manual ledger: `POST /finance/accounts/:id/balance`, `DELETE /finance/accounts/:id/balance/:entryId`.
  - Computed: `/finance/all`, `/finance/categorise?limit=N`, `/finance/monthly`, `/finance/variable-forecast?window=N`, `/finance/networth[?date=YYYY-MM-DD]`, `/finance/networth/history?months=N`, `/finance/projection?horizon=N&scenario=ID`, `/finance/budget-status?month=YYYY-MM`, `/finance/recurring/candidates`, `/finance/transfers/candidates`.
- **Net-worth historical reconstruction**: hub takes Monzo's live `total_balance` as "today" and walks cached transactions backward in time to reconstruct past balances (`balance(t-1) = balance(t) - delta`). Manual accounts use the ledger directly.
- **Investment growth** applies monthly to the investment portion of the projection at the rate from settings (default 5%/yr). A scenario's `investmentGrowth` delta overrides it.
- **Data-source reality check**: only Monzo is auto-synced. Lloyds/Revolut/S&S ISAs/GIAs/held-by-others are all "manual" accounts — log a balance entry whenever you check (Net worth tab → expand the row → add entry), runway is computed from those. Trading 212 / IBKR have APIs and could be wired later as a third account type if worth the bother; HL doesn't.
- **Scenario UX**: live in `src/components/money/ScenariosView.tsx` and CashflowView. Comparison chart overlays every saved scenario's liquid trajectory on the baseline. The "active scenario" picker on Cashflow highlights one for focused view; saved scenarios persist independent of active selection.
- **Charts**: `recharts` (~3.8). Tailwind 4 `var(--color-*)` tokens for axes/grid/tooltip so dark mode just works. `RunwayCard` is a 5-tile metric strip. `ProjectionChart` = liquid+total lines with emergency reference. `NetWorthView` history = stacked area liquid+investment.

### Meetup events subsystem (`server/src/meetup/`)

To fully remove a fork from the list, `con agent kill <session-id>` (`delete_session`). The old bug — `--end` leaving a running `<name> (fork)` (caught 2026-06-09; re-reported by the personal-migration agent) — was a **duplicate** hub session sharing the fork's `claudeSessionId` (a resume during a restart spawns one): `kill_session` killed only the resolved session, leaving the dup running, while `con agent kill`'s `delete_session` dup-guard cleared all. Fixed by giving `kill_session` the same dup-guard (mark all matching `claudeSessionId` ended, kept in list) + a `sessions_list` broadcast, and making `Session.kill()` set `status:ended` unconditionally (not only when a live process exists).
- **Manifest** (`~/.claude/console-hub-sessions.json`, written by `server/src/manifest.ts`): the list the hub uses to re-spawn sessions at startup. Every write is **synchronous and atomic** (`writeFileSync` to a `.tmp` sibling, then `renameSync`) — no in-memory debounce buffer, so a SIGKILL or host crash can't lose pending state. Every session with a `claudeSessionId` is persisted regardless of `status` (yes, including `ended`); removal is driven only by `delete_session`, not by subprocess lifecycle. This prevents the old failure mode where SDK timeouts / process exits would silently prune sessions from the manifest, so a later restart would resume zero. If you ever need to manually rebuild the manifest from JSONL transcripts on disk: the hub clobbers external writes within ms of any session event, so the only reliable path is **write the new manifest then `SIGKILL` the hub in the same Node tick** (the shutdown handler also calls `saveManifest`, so SIGTERM would still clobber); pm2/tsx will respawn the hub and it'll load the new manifest cleanly.

### Agent org chart — durable roles (`server/src/agents/registry.ts`)
Each agent is a **durable role** that outlives any live session, modelled on the Al pattern. State is an **agent-owned markdown file** `~/.config/console/agents/<key>.md`: frontmatter (`title`, `manager` = parent role's key, `goals`, `cwd`, `created`) + body (charter prose + a `## Memory` section the agent maintains). The **agent edits this file** with normal Read/Edit tools; the **hub reads it and never clobbers the body**.
- **`AgentRegistry`** (`server/src/agents/registry.ts`): boot-scans `agents/*.md` (parsed via the `yaml` dep — robust to comments/block-lists, unlike the hand-rolled `al/users.ts` parser), exposes `get/list/tree/resolveCharter/mintKey/create/setManager/delete`, and `fs.watch`es the dir (content-compared so the hub's own writes don't re-broadcast). `buildOrgTree()` is a pure, unit-tested fn that derives the org tree from `manager` edges and is defensive: a missing manager → annotated root (`danglingManager`), cycles broken with a visited-set (`cycleBroken`), nothing dropped.
- **Charter injection**: the role body becomes `--append-system-prompt` at the **`createSession` choke point** (`routes/agents.ts`) — only on a FRESH spawn (`!resume`, mirroring `session.ts`'s rule) and only when the caller didn't supply its own prompt (so Al keeps `buildAlSystemPrompt`). To apply an edited charter you must fresh-spawn — `con agent reload <name>` / the profile-panel "Reload" routes a role-backed session to `reloadAgentRole` (fresh spawn, charter re-derived; the `## Memory` carries forward). The **only field the hub writes is `manager`**, via a surgical single-line frontmatter stamp (`setManager` — never a `yaml.parse→stringify` round-trip, which would reorder keys / strip the agent's comments / reflow the body).
- **`agentKey`**: a stable, immutable slug (from the title), the join key across the role file ⇄ `ManifestEntry.agentKey` ⇄ `Session.agentKey` ⇄ `SessionInfo.agentKey`. Sessions resolve their role on spawn; a role with no live session is **parked** (revive = `reviveAgentRole` fresh-spawns it; ≤1 live session per role, enforced by a sweep mirroring the `delete_session` dup-guard).
- **Lifecycle**: `create_session{asAgent}` mints a role; `fork_session{seedRole}` mints a child role reporting to the source (`con agent chat` forks pass no `seedRole` → stay role-less). Ending a session **parks** the role (file stays); `delete_role` is the explicit removal (kills the live session + deletes the file; orphaned children become annotated roots, no cascade). **Al is the org root** (`agents/al.md`, `agentKey: 'al'`) but keeps its bespoke persona — `reload_session` special-cases `'al'` → `reloadAl`.
- **Backfill** (one-time, idempotent, in `index.ts` beside the fork-lineage backfill): turns every pre-existing named session into a role (title=name, manager=fork-parent's key via the shared `resolveParent`/`byName` two-pass, charter=initial prompt); writes only if the file is absent, stamps `agentKey` back to the manifest.
- **Surface**: protocol `agents_list{roles,tree}` (pushed on connect + every registry change) / `set_manager` / `revive_agent` / `get_agent_role` / `delete_role`; HTTP `GET/POST /agents/roles` (the out-of-band lever); CLI `con agent role list|get|manager|delete` + `con agent revive`.
- **UI** (`src/components/AgentTab.tsx`): a view-mode toggle (`agentViewMode`, localStorage `console:agents:viewMode`) flips the Agents pane between the session **list** and a visual **org chart** — `src/components/agent/AgentOrgChart.tsx`, a canvas node-link **tree** (`d3-hierarchy` `tree()` + `d3-zoom`, reusing the Notes circles-view canvas/zoom/drag engine; pure layout/hit-test in `agent-orgchart-helpers.ts`, unit-tested). **Folders are real role files** flagged `folder: true` (no session, drop targets, renamable). Interactions: left-click an agent → open its session, left-click a folder → its info; **right-click / long-press → context menu** (Show info, Open, Fork, Reload history/role, Revive, Rename, Delegate task…, Move to top level, Park, Delete — mirrors the list menu via a controlled `ContextMenuView`). Drag has a **6px threshold** so a jittery click isn't a reparent, and **drop-on-empty = cancel** (snap back), not make-root — drop on Al / "Move to top level" to go top-level. **Ctrl+Z / Ctrl+Y** undo/redo reparents + renames (store `orgPast`/`orgFuture`, scoped to the agents pane, ignored in inputs). A shared **"needs me" filter** (`filterAlerted`, store-backed) prunes the chart to the alerted subtree + ancestors. The role profile is a **modal** (`AgentInfoDialog` in AgentTab, opened via "Show info") not a bottom card; `AgentProfilePanel.tsx` shows charter/goals/`## Memory` (read-only — agent-owned) + a manager dropdown + revive/reload/park/delete (all guarded off for Al, the root).

### Agent delegation — org-aware task routing (`server/src/agents/{tasks,delegation,delegation-protocol}.ts`)
The org edges now carry work: Yousef tells **Al**, Al delegates DOWN the tree to the right agent, the result reports back UP through any intermediaries to Al, who relays to Yousef. Al stays the single point of contact and keeps his own session conversational — long work runs in the *delegated* sessions, never in Al's context.
- **Task** (`tasks.ts` `AgentTask`, file-backed at `~/.config/console/agent-tasks.json`, atomic tmp+rename like `model-config.ts`, terminal tasks pruned after 7d): `{id, title, brief, fromKey, toKey, origin, parentTaskId, chain[], status, result, workerSessionId, ephemeral}`. Deliberately minimal — NOT a Postgres ticket tracker.
- **Verbs** (`delegateTask`/`reportTask`/`runTaskWatchdog` live in `routes/agents.ts` beside `createSession`/`reviveAgentRole` to avoid a cycle; pure guards `checkDelegation`/`buildChain`/`chainLabel` in `delegation.ts`). `delegate` creates a task and **wakes the assignee's session** (the cron/`injectToAl` path: `broadcast(user_prompt) + logMessage + sendMessage`) with a self-instructing `[DELEGATED TASK]` envelope, then returns immediately (async, non-blocking). **The envelope lists the assignee's own reports and, when it has any, MANDATES routing onward** (exact `con agent delegate <reportKey> … --from <self> --parent <taskId>` command spelled out) so a manager can't short-circuit by answering on a report's behalf — it's generated fresh per delegation, so even a manager session that predates the protocol gets the routing instruction. Worker = the assignee's **durable** role session (`reviveAgentRole`, memory accrues, tasks queue) OR an **ephemeral** role-less worker spawned with the charter passed explicitly (`--ephemeral`, killed on report). `report` updates the task and wakes the **immediate delegator** (resolved by durable `agentKey`, survives reloads) with a `[REPORT]` envelope — managers synthesise and report up their own parent task (agent-driven, the hub does NOT auto-bubble). Guards: depth cap `MAX_DEPTH=6` + cycle check via `chain`.
- **Protocol preamble** (`delegation-protocol.ts` `buildDelegationProtocol()`): the single hub-owned "how to communicate" stanza, appended to **every** role's `--append-system-prompt` at the `createSession` choke point and to Al's `buildAlSystemPrompt`. Al's persona (`al/persona.ts`) adds the delegation-first + routing + reporting directives.
- **Permanent org position** (`buildOrgPosition()` + `renderOrgRoster()` + `shortDescription()`): every role spawn — including Al — gets injected at the same choke point: (a) the **full org roster** (everyone's NAMES as an indented tree, so any agent can *locate* anyone and see the path to them — fixes "Al can't find a deep agent"), plus (b) short **descriptions of just its immediate neighbours** (manager above + direct reports below, first charter sentence), plus (c) a **self-identity line** (`You are <title> (\`<key>\`) · role file ~/.config/console/agents/<key>.md`) so an agent knows its own key + file — load-bearing: it's how an agent locates its own file to self-maintain `## Memory`, and supplies the `--from <self>` key for delegating. Told **not to skip levels**. Fixed per spawn; a reparent applies on the next reload/revive (the role file's frontmatter `manager` is the live source of truth).
- **Watchdog**: a 5-min interval (`index.ts`) nudges in-progress tasks whose assignee went idle >15 min; after 2 nudges marks `blocked` and bubbles a stall report.
- **Fork branch-point seed**: a UI fork (`fork_session` with `seed:true`, set by the SPA `forkSession`) gets a `[FORK — branched from <parent>]` marker (`buildForkSeed` in `merge.ts`) so it knows where its own branch begins vs the inherited history — makes a later merge summary crisp. `con agent chat` forks pass no `seed` (they inject their own side-conversation seed). **GOTCHA (load-bearing): `claude --fork-session` emits NO `system` init — and so never hands the hub the forked `claudeSessionId` — until it receives a first message.** The seed is therefore sent **immediately** after the fork spawns, NOT on `session_init` (which would deadlock: silent fork → no message → no init → no csid → a dead, unusable fork — exactly the symptom when forking a large session). This is why `con agent chat` forks work (they message immediately) and why a `seed:false` UI fork would hang. Verified by spawning `claude --resume <csid> --fork-session` with vs. without stdin input.
- **Merge-back — fork OR org child** (`server/src/agents/merge.ts` pure builders + `mergeIntoParent`/`captureNextTurn` in `routes/agents.ts`; `mergeFork` is a back-compat alias): fold a child's knowledge into its parent instead of discarding it. The child self-summarises (`captureNextTurn` injects a merge-request and captures its next turn's text), the digest is injected into the parent, then the child is closed. A SUMMARY, not the transcript — keeps the parent's context clean. **Parent resolves two ways**: (a) **fork lineage** — `parentClaudeSessionId` → the live source session (`buildMergeRequest`, "you are a fork"); (b) **org manager edge** — the child role's `manager` → that role's live session, **revived if parked** so it can receive the digest (`buildChildMergeRequest`, "folding you into your manager"). For an org child the parent **absorbs the role too**: the child's own sub-reports reparent up to the parent, and the child's role file is deleted. `buildMergeEnvelope(name, summary, kind)` labels it fork vs agent. Guards: child must be idle; needs a fork-parent or a manager; no self-merge; if no summary is captured the child is left alive. Surface: protocol `merge_session`/`session_merged`; CLI `con agent merge <id|conv|name>` + `con agent chat --id <conv> --merge`; UI "Merge into parent" in the session list **and** org-chart context menu (shown for forks and any non-root org child). The reversible alternative to killing.
- **Hand-off**: Al emits `@handoff(<agentKey>)` (sentinel parsed by `server/src/handoff.ts`, mirrors `attention.ts`; scanned on finalized text in `session.ts`, emits `session_handoff`). The SPA shows an opt-in "Talk to X" banner (store `pendingHandoff` → `acceptHandoff`) + a "↩ Back to Al" control (`handoffReturnTo`); the raw sentinel is stripped from Al's displayed text. **WhatsApp/voice hand-off is out of scope** — Al relays there.
- **Surface**: protocol client `delegate`/`report`/`cancel_task`/`tasks_list`, hub `tasks` (pushed on connect + change) + `session_handoff`; HTTP `GET/POST /agents/tasks`, `POST /agents/tasks/:id/report`, `DELETE /agents/tasks/:id`; CLI `con agent delegate|report|tasks`. SPA: org-chart **task badges** (violet ↓ + count per assignee) + **active-delegation edges** (org edges on an open task's `chain` render yellow with animated marching dashes — `activeEdges` in `AgentOrgChart.tsx`), `TasksPanel.tsx` (toggle in the pane header), tasks in `src/store/agent.ts`.
- **Out of scope** (deliberately not paperclip): Postgres/ticket-UI/budgets/SLAs; cross-machine routing; hub auto-bubbling (managers bubble agent-driven, by design).

### Hub-side agent cron (`/cron/*`)
Claude Code's built-in `/loop` / `CronCreate` is session-scoped and DOES NOT survive hub restarts under our SDK transport (`--input-format stream-json`) — empirically verified, matches Anthropic bugs [#40228](https://github.com/anthropics/claude-code/issues/40228) (`durable: true` doesn't persist) and [#40081](https://github.com/anthropics/claude-code/issues/40081) (sessions with cron tasks vanish from `/resume`). So the hub runs its own.
- **Scheduler**: `server/src/cron/scheduler.ts` (Croner + debounced JSON persist to `~/.config/console/agent-cron.json`). Tasks keyed by `claudeSessionId` (stable across restarts; hub session ids are rejected). Forks get a new claudeSessionId so they inherit zero tasks. Fires by mirroring the post-restart "Continue." nudge path: `broadcast(user_prompt) + session.logMessage + session.sendMessage`. Auto-disables a task after 10 consecutive misses (session not found / ended). One-shot tasks self-remove after firing. Croner `protect: true` prevents overlapping fires when Claude takes longer than the cron interval to respond.
- **Routes** (`server/src/routes/cron.ts`): `GET /cron[?session=…]`, `POST /cron`, `DELETE /cron/:id`, `POST /cron/:id/run`, `GET /cron/upcoming?n=N`, `GET /cron/ics-token` (returns `{token, publicUrl}`), `GET /cron.ics?token=…` (constant-time compare; 403 on bad token; `Cache-Control: max-age=300`).
- **ICS**: `server/src/cron/ics.ts` expands each task into discrete VEVENTs (next 50 firings, 30-day window) via `Croner.nextRun()` — avoids the lossy cron→RRULE conversion. UTC timestamps, RFC5545 line folding + escaping. Calendar clients re-fetch periodically.
- **CLI**: `con cron list/add/remove/run/upcoming/ics-url`. `--trigger` accepts a 5-field cron, ISO datetime, or `+30m`/`+2h`/`+1d` (CLI resolves relative to ISO). `--once` flips recurring=false (implied for ISO/relative).
- **SPA**: `src/components/agent/CronPill.tsx` (status-bar chip) + `CronPanel.tsx` (side panel: list, run-now/delete, recurring/one-shot create form with live validation, ICS subscription URL with copy). Backed by `src/store/cron.ts` (polled, no WS sync in v1).
- **Public exposure for GCal**: Caddy serves the ICS at `https://con.amar.io/public/cron.ics?token=…` (the `/public/*` namespace is in the auth allow-list). Legacy `/cron.ics` is Caddy-rewritten to the new path so existing GCal subscriptions keep working. `getPublicIcsBase()` returns `CONSOLE_PUBLIC_ORIGIN` (default `https://con.amar.io`). Token query string is the only auth — don't share it.

### Push-to-talk (PTT) — system-wide hold-to-talk → STT → mic owner
A hardware/hotkey hold-to-talk that streams mic → hub `/stt` (OpenAI realtime, `{type:'audio',data:b64}` → `{interim|final,text}`) and on release routes the transcript to the **mic owner** session. Owner model in `server/src/mic.ts` (`MicState`: `ownerSessionId` persisted to `mic-owner.json`, transient `hot`); routes in `server/src/routes/mic.ts` (`/mic/status|owner|hot|say|compose`); SyncBus `mic` service in `index.ts` broadcasts `state` (owner/hot) + `compose` (transient, NOT in the subscribe snapshot — and the SyncBus never replays missed events). Default owner = explicit-if-live-else-Al. CLI `con mic status|give|pass|release|say|ptt`. SPA mirror in `src/store/mic.ts` + the `MicButton` in `AgentTab`.
- **Two delivery modes**: `/mic/say` = inject + **auto-send** to the owner; `/mic/compose` = drop into the owner's composer **unsent** for review (`AgentPromptInput` reacts to `composeSeq`). A composed draft **lingers** in the textarea until sent/cleared — by design (it's a review draft), so an earlier foreground compose can still be sitting there later; it is NOT a duplicate of a backgrounded `say`.
- **Desktop** (Sway, out-of-repo dotfiles): `$direct+c`/`$remote+c` hold → `~/exec/con-ptt` → `parec --rate=24000 … --latency-msec=50` → `/stt` WS, live transcript in a `foot` popup (`/tmp/con-ptt.live`), routes to `/mic/compose` (review) on release; waybar `custom/ptt` indicator (`/tmp/con-ptt.hot`).
- **Phone** (APK v26+): the rugged phone's Custom-key **"Open Zello"** Easy-shuttle mode is the ONLY one that emits the firmware-level `com.zello.ptt.down/.up` broadcasts (Zello PTT API action strings — independent of any app). `PushService.pttReceiver` records on `.down` (AudioRecord VOICE_RECOGNITION 24kHz mono PCM16 → `/stt`), and on `.up` routes to `/mic/compose` if `MainActivity.foreground` else `/mic/say`. Needs FGS type `microphone` + RECORD_AUDIO. The native service requires the **apk bearer** (pair the APK) — see [Known Issues] + `memory/project_android_js_bridge_bind.md`.
- **`android/zello-shim/`** — a tiny invisible stub app with applicationId **`com.loudtalks`** (Zello's REAL package — `com.zello` is only the broadcast prefix). "Open Zello" deep-links to the Play Store when `com.loudtalks` is absent; installing this no-op (a translucent activity that finishes instantly) makes the firmware's launch resolve to nothing visible while the PTT broadcasts keep firing. Build `./gradlew :zello-shim:assembleDebug`, served at `/public/apk/loudtalks-shim.apk`.

### Android APK
Requires Android SDK at `$ANDROID_HOME` (default `~/app/Android/Sdk`), minSdk 26, targetSdk 35. Applicationid `io.amar.console`.
- **Always build release, never debug.** Single-user software — the in-app updater (`/apk/latest.json`) only serves the release channel, debug builds can't auto-update, and there's no QA/staging lane that benefits from a separate debug variant.
- `android/scripts/generate-keystore.sh` — one-time, create `~/.config/console/console-release.jks` (back it up!)
- `android/scripts/build-release.sh` — build signed release, copy to `~/.config/console/apk/`, write `latest.json` (reads `~/.config/console/apk-release.env` for passwords). Phone picks up the update on next launch via the in-app banner.
- `android/scripts/build-debug.sh` exists but should not be used.
- The APK loads `https://con.amar.io/` since v21 — fully off-tailnet, Caddy serves SPA + hub + public surface from the same origin. v18-v20 used a Tailscale Funnel host and are obsolete (legacy `/apk/*` Caddy rewrite keeps stuck installs upgradable). HMR still works through Caddy's reverse_proxy to Vite.
- Web side: `isNative()` (from `src/platform.ts`) detects `window.__isConsoleAPK`. Used by `src/gmail/auth.ts` (passes `?callback=app` so hub redirects OAuth to `console://auth/done`) and `src/main.tsx` (requests `navigator.storage.persist()`).
- Remote debugging: `chrome://inspect` on the laptop while phone is USB-connected (debug builds only; `setWebContentsDebuggingEnabled(BuildConfig.DEBUG)`). The embedded debug agent also works — APK events show in `/debug/log` with UA containing `ConsoleAPK/...`.
- Push notifications: `PushService` opens a persistent WebSocket to `/push` and posts system notifications even when the WebView is backgrounded. Server sources call `pushServer.broadcast({ type, title, body, pane, id, ...chatFields })`. Wired today: Monzo webhook (`money` channel), agent `AskUserQuestion` / `ExitPlanMode` (`agent` channel), Gmail new mail (`mail/sync.ts` — fires on `messagesAdded` deltas), Matrix room messages (`matrix/sync.ts` — see below), Calendar per-event reminders (`cal/sync.ts` — 30s ticker scans upcoming events, fires on `reminders.overrides` minutes-before-start with a 60s slack window, dedupes via persisted `fired` map). CLI/webhooks can emit via `POST /push/send`. Check connected clients: `curl -sk https://localhost:9877/push/status`.
  - Chat push specifics (`server/src/matrix/sync.ts`):
    - **Mute filtering**: skip rooms where the delta's `unread_notifications.notification_count === 0` (server-side push rules muted it) OR where the room is tagged `m.lowpriority` / `m.archive` with no highlight. Mirrors the browser's unread logic.
    - **Enrichment**: `PushMessage` carries `roomId`, `roomName`, `senderName`, `senderId`, `senderAvatarMxc`, `roomAvatarMxc`, `isDirect`, `timestamp`. **`roomName`/`isDirect`/`roomAvatarMxc` are read from the canonical, persisted `ChatRoomsStore` snapshot (`chat-rooms.json`), NOT the local in-memory `roomState` cache.** Why: the local cache only learns a room's name from `m.room.name` state events, which Matrix replays on initial sync (or when the name changes). After a hub restart, an incremental sync delivering a group message carries no state events, the local cache stays empty, and the notification fell back to the **sender's** name — so every group looked like a 1:1 DM. The `ChatRoomsStore` resolves names via heroes/bridge-member fallbacks (`computeRoomState`/`getRoomName` preserve the prior resolved name across nameless deltas) and survives restarts on disk, so group titles are always correct. The local `roomState` cache (name, avatar mxc, members with displayname+avatar) + `directRooms` set (from global `m.direct` account_data) are still maintained per sync tick + from the `snapshot` RPC, and still source per-event `senderName`/`senderAvatarMxc` from `members`; they're just the fallback for name/isDirect/avatar now.
  - APK rendering (`PushService.kt`):
    - Chat notifications use `NotificationCompat.MessagingStyle` with `Person` + circular avatar (fetched once via hub's `/matrix/media/thumbnail/...` proxy, cached in-memory). History keeps the last 8 messages per room.
    - Grouping: `setGroup(CHAT_GROUP_KEY)` + a single summary notification (id 100). Per-room notifId is `roomId.hashCode()` so follow-up messages update the same card.
    - Vibration debounce: max one vibration per room per 60s — subsequent messages build with `setSilent(true)` + `setOnlyAlertOnce(true)`.
    - Deep link: `console://pane/chat?roomId=<id>`. `MainActivity.handleDeepLink` parses the query param and dispatches `console:navigate` with `{ pane, itemId }`; `src/notifications.ts` listens for that event and calls `handleNotificationClick` (same path as browser notifications), so tapping a chat notification lands in the right room.
- APK update check: runs once in `onCreate` (cold start) comparing `/apk/latest.json` versionCode against `BuildConfig.VERSION_CODE`. The refresh button in `src/components/Layout.tsx` also triggers a re-check by calling `window.ConsoleNative.checkForUpdate()` — bridge exposed via `addJavascriptInterface(ConsoleBridge(), "ConsoleNative")` in `MainActivity.buildWebView`. No-op in the browser. `updateBanner` tracked so repeated calls don't stack banners.

## Hub Routes
External clients reach all of these under `${origin}/hub/<path>` via Caddy (the prefix is stripped before the upstream sees it). Local-host clients (CLI, Al) hit `https://127.0.0.1:9877/<path>` directly with a bearer. The `/public/*` and `/auth/google/*` paths are the only ones the auth middleware lets through unauthenticated.
- `/auth/*` — Google + Monzo + Matrix OAuth flows.
  - `/auth/session` (GET) + `/auth/session/logout` (POST) — current hub session (cookie-derived).
  - `/auth/hub/sessions` (GET) — list active browser sessions (id-prefixes only).
  - `/auth/hub/tokens` (GET/POST), `/auth/hub/tokens/:id` (DELETE) — name + scope-keyed bearer tokens for non-browser clients. POST returns the plaintext once.

## Al (absorbed into Console hub)

The standalone Al daemon (`~/proj/code/al/`) was retired 2026-06-09. Its WhatsApp + voice + persona runtime now lives in-process at `server/src/al/` and is owned by Console hub. Al himself is a permanent Console-managed Claude Code session (no more `pi-agent-core` loop). Persona (`AL.md`, `mistakes.md`, workflows summary) is loaded as the system prompt **only on a fresh spawn** (`--append-system-prompt`, applied just once — never re-appended on resume or model-respawn; see `session.ts:195`). To re-derive the persona after editing `AL.md`/`persona.ts`/etc., run **`con agent reload Al`** — no hub restart needed. It maps to the `reload_al` client message → `reloadAlSession()` (`al/al-session.ts`): kills the current Al, clears `al-session.json`, re-runs `ensureAlSession` → fresh spawn with a new `claudeSessionId` and freshly-derived persona. Works whether Al is up or already down. (A plain hub restart only *resumes* Al with his old baked-in prompt; and editing `server/src/al/*` source still needs `con hub restart` to deploy — but the persona itself then applies via `con agent reload Al`.) For non-Al sessions, **`con agent reload <id|name>`** → `reload_session` → `Session.reload()` respawns the subprocess via `--resume` (history preserved) — the lever to recover a stuck/dead session without bouncing the hub. No fs.watch / no live reload. (There is no `con agent restart`; reload is the verb.)

### Inbound flow
Baileys `onMessage` (after `fromMe` filter) → `ensureUserKnown` → `inboundEnvelope(msg, resolvedUser)` → `injectToAl(envelope, broadcast)`. The envelope is **task-framed** (not a chat message) so Claude doesn't default to a silent in-session reply. Shape:
```
[INBOUND WhatsApp — action required]
From: <senderName> (<sender-jid>) — resolved user: <name or "unknown">
Thread: <jid>
Message ID: <wamid.HBgL...>

Message:
<body>

[Attached image(s) (Read these paths): /tmp/console-al-inbound/<id>-0.jpg]   ← only when an image is attached

---
ACTION ... con whatsapp send <jid> --body "..."  (a session-only text reply is invisible to the sender)
```
Inbound images are downloaded by `whatsapp.ts`, written to `os.tmpdir()/console-al-inbound/`, and their paths injected into the envelope so Al can `Read` them (the text-only inject path can't carry image bytes). Al's Claude context IS the routing state — no per-thread session table. Voice delegate envelopes use a `[Voice delegate from ...]` header with explicit "reply will be spoken aloud" guidance.

## Matrix recovery workflow (hub-side)
The old browser "Settings → recovery key" flow was dropped in the hub-centric migration. Hub-side replacements live in `server/src/matrix/backup-restore.ts` + `secret-storage.ts`, exposed as:
- `POST /matrix/keys/restore-from-recovery-key {recoveryKey}` — decodes an `EsU…` key; tries direct `BackupDecryptionKey` match first, then SSSS unlock. SSSS path iterates **every** keyId that encrypts `m.megolm_backup.v1` — users often have the current default SSSS key point somewhere new (e.g. `RII4BrO2Ox`) while the backup secret is still encrypted under the previous key (e.g. `bXFaNVGpGwtYSl7WxkfM2aznevIvcl3z`). Decrypts all sessions via `/room_keys/keys` and imports into OlmMachine (+ persists via `saveBackupDecryptionKey`).
- `POST /matrix/keys/restore-cross-signing {recoveryKey}` — decrypts the three cross-signing private secrets (`m.cross_signing.{master,self_signing,user_signing}`) from account_data, calls `importCrossSigningKeys` then `bootstrapCrossSigning(false)`, and explicitly sends the returned device-signature upload (bootstrap requests do NOT flow through `outgoingRequests()`). Required after any hub re-login — bridges reject Olm from uncross-signed devices. Beeper's current default SSSS key `RII4BrO2Ox` ships with **no `iv`/`mac` verification block**, so the helper skips `verifySsssKey` when those are absent and relies on the secret's own MAC to reject wrong keys.
- `POST /matrix/keys/import-local-backup` — force-imports the on-disk M0 key backup into the hub's current OlmMachine. Useful after hub migrates to a new deviceId and the legacy browser-exported keys never got merged.
- `POST /matrix/decrypt-event {roomId, eventId|event}` — fetches raw event from the homeserver (or takes it inline) and decrypts via the hub's OlmMachine. Used to resurrect "🔒 Encrypted message" placeholders that the browser persisted to IDB when the hub couldn't decrypt them at sync time.

Multiple rotations of the recovery key are fine — keep older keys handy because `tryKeyAgainstVersion` walks **all** backup versions and the SSSS loop walks **all** keyIds, so any historical key that unlocks any of them still restores. Beeper's recovery keys (`EsU…` prefix) are SSSS keys, not direct `BackupDecryptionKey` seeds. Cross-signing secrets are often encrypted under a **different** (newer) SSSS key than megolm backup — the two routes accept different recovery keys on the same account.
